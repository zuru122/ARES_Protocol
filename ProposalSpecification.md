## Ares Protocol Specification

### Overview
Ares is a treasury execution system built accross four (4) independent modues, which handles all funds movement, hence, all funds most pass all this fourstages: which includes:

- proposal
- authorization
- time delay
- execution

#### Proposal Creation
In Proposal Creation, anyone can propose treaury action by calling `creatProposal()`on the Proposal managemnt `ProposalMg`. they must have 0.1 ETh as a deposit to prevent spaming.

During creation, a unique ID is generated by hashing all proposal fields together. he proposal cannot move forward for at least 1 hour (commit phase).

#### Authorization
Proposal must be approved by 2 out of 2 signers "based on my implementation" using the EIP-712 signatures after the commit phase. Each signature is tied to:
- The specific proposal address
- The signer's current nonce
- A deadline timestamp
- This chain and this contract
Once the signatures are verified, the nonces increment permanently, hence, the same signature can never be reused.

#### Queueing
Once authorized, `queueProposal()` moves the proposal to QUEUED. It then enters the `Timelock` where a 48-hour countdown begins.
During this window:

The proposal cannot execute
Governance can still cancel it
The community can observe and react incase any malacious movement is detected.

#### Execution
After 48 hours that's been set,  execute() function can be called. The system automatically checks:
- entry exists?
- status == QUEUED?
- 48 hours passed?
- daily limit ok?
State updates to `EXECUTED` before the external call. This blocks reentrancy. The `Timelock` then calls `AresProtocol.executeProposal()` which makes the actual treasury call. AresProtocol rejects any caller that is not the registered timelock.


#### Cancellation
A proposal can be cancelled while still on `PENDING` or `QUEUED` by:

The original proposer,  deposit refunded
An authorized signer, deposit slashed due to malicious act detected by the Governance.

Once `EXECUTED` it is permanent and cannot be reversed.

#### Reward Distribution
The reward list is compiled off-chain by the governance, whereby it hashes each contributo and their amount into a leaf, then combines all leaves into a Merkle tree. when claiming, each contributor only needs to submit therir address, amount and proof. The contract rebuilds the path and checks if it reaches the stored root. if it matches, tokens are sent and the address is permanently marked as claimed. It also ensure that an address can't claim twice.
